# 0.1 Benchmarking Design

## Status

Draft. Do not treat this as implemented API.

## Goal

Add a generic benchmarking layer for PocketIC-based end-to-end tests by parsing structured profiling marks emitted by canisters into logs. The feature should help downstream projects measure instruction and memory deltas without making `ic-testkit` own application-specific benchmarks, labels, thresholds, or CI policy.

The motivating proposal came from an SR9P-style marker format. `ic-testkit` should support that shape as a compatibility option, but the implementation should frame this as a generic log-marker protocol. The default `ic-testkit` marker prefix is `ICTK`.

## Non-Goals

- Do not replace `pocket-ic`.
- Do not benchmark host wall-clock time around PocketIC calls as the primary signal.
- Do not define application benchmark suites, canister labels, regression thresholds, or pass/fail policy.
- Do not require a specific canister framework or generated endpoint model.
- Do not assume benchmark labels map to a particular app topology.

## Marker Model

Canister code emits profiling marks by printing structured lines to canister logs. PocketIC normally prints those debug/log lines into the test process stdout/stderr, so the E2E harness should capture stdout/stderr and scan that captured text after each spec.

The first marker shape should be compact and positional because benchmark marks may be emitted many times in a run:

```text
ICTK|<label>|<instructions>|<heap_bytes>|<memory_bytes>|<total_allocation>
```

Example lines:

```text
ICTK|mycan/myfunc/anything:start|23423234|234234234|234232|234234
ICTK|mycan/myfunc/anything:end|23455210|234239900|234240|234990
```

The prefix is configurable. `ICTK` is the default prefix for this crate. Downstream projects can opt into their own compact prefix, such as `ICY`, when they want project-local logs. `SR9P` can be supported as a compatibility prefix for imported logs or downstream experiments, but the parser must not bake in SR9P project semantics.

No field names are included in the 0.1 compact marker. If a future version needs optional fields such as `span_id`, `scope`, `canister_id`, or `spec`, it should add an explicitly versioned extended marker instead of changing the meaning of this tuple.

The benchmark suite is derived from the label by default. The initial rule should use the first path segment before `/`, so `mycan/myfunc/anything:start` belongs to suite `mycan`. A parser config may override suite derivation if a downstream harness wants a different grouping.

## Canister Emitter Helper

`ic-testkit` should provide a tiny canister-side helper so downstream canisters do not hand-format marker lines:

```rust
use ic_testkit::performance::Performance;

Performance::measure("app/myfunc/something:start");
Performance::measure("app/myfunc/something:end");
```

The helper should:

- read the current instruction and memory counters from the IC CDK;
- print exactly one compact marker line in the configured protocol shape to the canister log;
- avoid app-specific label rules beyond requiring `:start` or `:end`;
- avoid thresholds, benchmark definitions, or CI policy;
- be feature-gated if adding IC CDK dependencies would otherwise affect host-only `ic-testkit` users.

Sketch:

```rust
pub mod performance {
    pub struct Performance;

    impl Performance {
        pub fn measure(label: &str) {
            // Reads IC/CDK counters, then prints:
            // ICTK|<label>|<instructions>|<heap_bytes>|<memory_bytes>|<total_allocation>
        }
    }
}
```

The exact IC CDK calls need to be verified during implementation, but the public surface should stay as small as `Performance::measure(label)`.

## Marker Grammar

The 0.1 grammar is intentionally simple:

```text
marker = prefix "|" label "|" instructions "|" heap_bytes "|" memory_bytes "|" total_allocation
prefix = configured ASCII prefix
label = UTF-8 string without "|", "\n", or "\r"
counter = ASCII decimal unsigned integer
```

No escaping is supported in 0.1. Labels containing `|` or newlines are malformed. Percent-encoding or another escaping strategy can be added later if real labels need it.

Marker validation:

- `prefix` must match a configured prefix.
- `label` is required and must be nonempty.
- `label` must end in exactly one of `:start` or `:end`.
- marker lines must have exactly six pipe-separated columns.
- all four counter columns are required.
- all four counters must parse as unsigned integers.
- non-marker lines are ignored and may be counted for diagnostics.

## Event Fields

Required fields:

- `prefix`
- `label`: benchmark label ending in `:start` or `:end`
- `suite`: derived suite or spec group
- `instructions`
- `heap_bytes`
- `memory_bytes`
- `total_allocation`

Reserved future fields for an extended marker:

- `span_id`: explicit span identifier for pairing repeated or interleaved spans.
- `scope`: execution-counter scope identifier.
- `execution_id`: alias-level concept for execution scope if downstream emitters prefer that term.
- `canister_id`: canister principal that emitted the mark.
- `method`: canister method or logical operation name.
- `spec`: test spec name.

The compact 0.1 marker has no unknown fields. A future extended marker should preserve unknown fields in raw event output and ignore them by default aggregation unless configured otherwise. Malformed numeric counters should make the event malformed rather than silently coercing to zero.

## Pairing Rules

The parser splits `label` into:

- `span_label`: label without the `:start` or `:end` suffix
- `kind`: `start` or `end`

Pairing is deterministic:

1. Pair compact 0.1 markers by stack order per `(suite, span_label)`.
2. If a future extended marker includes `span_id`, pair by `(suite, span_label, span_id)` before falling back to stack order.

Stack pairing handles simple repeated and nested spans:

- `A:start`, `A:end`
- `A:start`, `A:start`, `A:end`, `A:end`

The parser must record:

- paired spans
- unpaired starts
- unpaired ends
- malformed marker lines
- invalid spans
- non-marker lines ignored by the parser

Unpaired, malformed, and invalid records should appear in the report. CI failure policy belongs to the downstream harness.

## Invalid Spans

Malformed markers are syntax or field-shape failures. Invalid spans are semantic failures after otherwise valid markers are paired.

Invalid span examples:

- end counter is lower than start counter
- future extended start/end markers disagree on immutable fields such as `scope`, when both are present
- future extended marker includes `scope` or `execution_id` on only one side
- future extended paired markers cross a declared execution scope

The first implementation should write invalid spans separately from malformed markers and unpaired markers.

## Counter Semantics

Counters are unsigned integer values. Use `u128` internally for parsed counters, deltas, and aggregate totals so large suites do not overflow ordinary host-sized integers.

Span deltas are computed as:

```text
end.counter - start.counter
```

Negative deltas are invalid spans, not wrapping arithmetic.

Instruction deltas are meaningful only when the start and end marks belong to compatible execution-counter scopes. Downstream emitters should avoid spans that cross message executions unless they understand the counter semantics they are using.

Scope behavior:

- Compact 0.1 markers do not include scope information, so compute deltas but report `scope_status=unknown`.
- If a future extended marker includes `scope` on both start and end, they must match.
- If a future extended marker includes `execution_id` on both start and end, they must match.
- If a future extended marker includes `scope` or `execution_id` on only one side, the span is invalid.

CSV should serialize counter values and deltas as decimal strings if needed to avoid consumer integer-width surprises. Internally they remain `u128`.

## Harness Flow

1. Run each E2E spec under PocketIC.
2. Capture stdout/stderr for the full spec run.
3. Scan captured text for marker-prefix lines.
4. Parse raw marker events.
5. Pair start/end events into spans.
6. Compute deltas for `instructions`, `heap_bytes`, `memory_bytes`, and `total_allocation`.
7. Separate valid spans from invalid spans.
8. Aggregate by suite.
9. Aggregate across `ALL` suites.
10. Write raw events, paired spans, invalid spans, aggregate CSVs, and a Markdown analytics summary.
11. Optionally compare the current aggregate rows against a selected previous run.

## Log Capture Strategy

The first implementation should prefer process output capture over direct `actor.get_logs()`-style fetching. The working assumption is that normal canister debug lines produced during PocketIC calls are already printed to stdout/stderr by the test process.

This matters because PocketIC canister logs may be trimmed after a small number of lines, for example around 200 lines. The harness should therefore capture output while the calls are running rather than relying on a late query against an in-memory log buffer after many calls have completed.

Direct log fetching should be optional and evidence-driven:

- use stdout/stderr capture as the default source;
- only add direct log fetching if tests show a scenario where stdout/stderr misses benchmark markers;
- if direct fetching is added, perform it immediately after relevant calls to avoid trimmed logs;
- record the source of each raw event as `stdout`, `stderr`, or `fetched_log`.

Known scenarios to verify before implementation is considered complete:

- ordinary update/query calls with debug output;
- many emitted lines where an in-memory log buffer may trim older entries;
- canister timers or heartbeat-style execution;
- calls that trap or return errors after emitting a start marker;
- concurrent or repeated specs where output capture could interleave lines.

## First-Release Report Layout

The 0.1 implementation should keep output small and CSV-first.

Reports should be written under a stable run directory that carries enough identity for agents and humans to understand which code produced the numbers:

```text
reports/runs/<date>-<short-commit>-<index>/
```

Example:

```text
reports/runs/2026-05-24T162600Z-a1b2c3d-0001/
```

The index disambiguates repeated benchmark runs for the same commit and timestamp bucket. The writer should also record the full commit hash in `metadata.json`. If git metadata is unavailable, the writer should use an explicit `unknown` commit segment and still write metadata explaining that limitation.

Required files:

```text
raw-events.csv
spans.csv
suite-aggregates.csv
all-aggregates.csv
malformed-markers.csv
unpaired-markers.csv
invalid-spans.csv
bench-summary.md
metadata.json
```

Deferred or optional files:

```text
reports/latest-bench-summary.md
reports/latest-spans.csv
reports/latest-suite-aggregates.csv
reports/latest-all-aggregates.csv
```

JSONL can be added later if a downstream workflow needs it. The first version should keep durable machine-readable output to CSV plus `metadata.json`.

## Analytics Summary

The Markdown summary should be optimized for the main consumer goal: an agent can see how much each measured operation costs and how that cost changed against the previous test suite run.

The primary table should be grouped by benchmark label and show runs, average cost, and comparison percentages. Example shape:

```markdown
| Benchmark | Runs | Instructions Avg | Heap Delta Avg | Memory Delta Avg | Allocation Avg |
| --- | ---: | ---: | ---: | ---: | ---: |
| app/myfunc/something | 12 | 0.2342B (+34%) | +1.2 MB (+8%) | +4.0 MB (-23%) | +0.8 MB (+2%) |
```

Formatting rules:

- instruction deltas should be displayed in billions with a `B` suffix, for example `0.2342B`;
- memory-like deltas should be displayed in human units, for example `+4.0 MB`;
- comparison percentages should compare the current average against the selected previous run;
- positive and negative signs should be explicit for deltas and percentage changes;
- missing previous rows should be shown as `new`;
- missing current rows should be reported in a separate removed-benchmarks table.

The summary can include totals and peaks after the primary table, but the first screen of the report should answer: what ran, how many times, what it cost on average, and how that changed.

## Aggregates

For each `(suite, span_label)` and for `ALL`, compute:

- count
- total delta per counter
- average delta per counter
- min delta per counter
- max delta per counter
- peak observed end counter, where meaningful

The first implementation should keep aggregates simple and CSV-friendly. The Markdown analytics summary may format these same values for readability, but CSV files should keep raw numeric values and separate comparison columns.

## Comparison

Comparison helpers are useful, but thresholds and CI failure behavior remain downstream policy.

When a previous test suite run exists, compare current aggregate rows against that run by stable key:

```text
suite + span_label
```

Report:

- added rows
- removed rows
- absolute delta changes
- percentage delta changes
- current value
- previous value

For averages, percentage change is:

```text
((current_avg - previous_avg) / previous_avg) * 100
```

If the previous value is zero, the report should avoid division by zero and display the percentage as unavailable.

Previous-run selection should be explicit in APIs. A convenience selector may choose the newest previous run by:

1. scanning `reports/runs/<date>-<short-commit>-<index>/`
2. requiring parseable `metadata.json`
3. ignoring malformed run directories
4. sorting by timestamp
5. excluding the current run
6. choosing the latest prior run for the same benchmark suite when a suite name is recorded
7. otherwise choosing the latest prior successful parse

Record the comparison basis in metadata:

```json
{
  "comparison": {
    "previous_run": "reports/runs/2026-05-24T155200Z-9f8e7d6-0001",
    "key": ["suite", "span_label"]
  }
}
```

## Metadata

Each run should capture enough context to make results comparable:

- timestamp
- run directory name
- run index
- git commit hash
- git commit short hash
- `ic-testkit` version
- `pocket-ic` version
- rustc version
- wasm build target/profile arguments
- selected feature flags
- host OS and architecture
- parser prefixes
- strict-mode setting
- suite-derivation setting
- benchmark command or suite name when provided
- selected previous run when comparison is performed

## Strict Mode

Strict mode should not change report generation. It changes the high-level runner result after reports are written.

Recommended strict-mode behavior:

- malformed markers make the high-level run return an error
- unpaired markers make the high-level run return an error
- invalid spans make the high-level run return an error
- reports are still written before returning the error

Non-strict mode records these problems but returns success if the benchmark command itself succeeded.

## Proposed Public API

Keep the core pure and testable. File-system reporting should be a thin layer.

Sketch:

```rust
pub struct BenchmarkParserConfig {
    pub prefixes: Vec<String>,
    pub suite_derivation: SuiteDerivation,
    pub strict: bool,
}

pub struct RawBenchmarkEventV1 {
    // parsed compact marker, derived suite, counters, source line/source stream metadata
}

pub struct BenchmarkSpanV1 {
    // start/end events, deltas, scope status
}

pub struct InvalidBenchmarkSpanV1 {
    // paired events plus invalid reason
}

pub struct BenchmarkParseReportV1 {
    // raw events, malformed markers, ignored line count
}

pub struct BenchmarkAggregateRowV1 {
    // suite, span_label, count, counter summaries
}

pub struct BenchmarkRunMetadataV1 {
    // run context and parser/report config
}

pub struct BenchmarkReportFormatConfig {
    // human units, instruction precision, comparison display rules
}

pub fn parse_benchmark_events(
    input: &str,
    config: &BenchmarkParserConfig,
) -> BenchmarkParseReportV1;

pub fn pair_benchmark_spans(
    events: &[RawBenchmarkEventV1],
    config: &BenchmarkPairingConfig,
) -> BenchmarkSpanReportV1;

pub fn aggregate_benchmark_spans(
    spans: &[BenchmarkSpanV1],
    config: &BenchmarkAggregateConfig,
) -> BenchmarkAggregateReportV1;

pub fn compare_benchmark_aggregates(
    current: &[BenchmarkAggregateRowV1],
    previous: &[BenchmarkAggregateRowV1],
) -> BenchmarkComparisonReportV1;

pub fn write_benchmark_report_dir(
    report: &BenchmarkRunReportV1,
    path: impl AsRef<Path>,
) -> Result<()>;
```

Exact types can change before implementation, but the architectural split should remain:

- parse
- pair
- aggregate
- write reports

## Proposed `ic-testkit` Boundary

`ic-testkit` can own:

- minimal canister-side marker emitter helper
- stdout/stderr capture helpers for benchmark runs
- marker parser
- event and span data model
- pairing logic
- aggregate calculations
- CSV and Markdown report writers
- metadata writer
- previous-run comparison helpers
- analytics summary formatter

Downstream projects should own:

- where and when marker emission is called inside canister code
- benchmark suite definitions
- labels and label taxonomy
- thresholds
- CI failure policy
- where to fetch logs from if they use custom harness plumbing or need non-stdout sources

## Required Tests Before Implementation Lands

- non-marker lines ignored
- canister helper emits compact marker shape
- stdout capture records markers from ordinary PocketIC calls
- stderr capture records markers when PocketIC writes there
- late direct log fetching is not required for ordinary debug lines
- many log lines do not lose captured stdout/stderr markers even if PocketIC's internal log buffer trims older entries
- timer or heartbeat-style marks are captured or documented as requiring direct fetch support
- trap/error calls that emit a start marker still record the marker as unpaired
- concurrent or repeated specs do not corrupt line boundaries during capture
- valid start/end pair
- repeated same-label spans
- nested same-label spans
- stack pairing per `(suite, span_label)`
- unpaired start
- unpaired end
- malformed numeric counter
- malformed too few columns
- malformed too many columns
- negative delta becomes invalid span
- empty `label`
- invalid label suffix
- marker prefix with missing columns
- suite derived from label
- deterministic ordering of spans and aggregates
- configurable prefix
- `ICTK` default prefix behavior
- `SR9P` compatibility prefix behavior
- previous-run comparison added/removed/changed rows
- previous-run selection by suite metadata
- run directory name includes date, short commit, and index
- unknown git metadata falls back to an explicit unknown segment
- Markdown analytics table includes runs, formatted averages, and comparison percentages
- instruction averages format in billions
- memory averages format in human units
- zero previous value does not divide by zero
- CSV escaping
- metadata contains versions and parser config

## Open Questions

- Should malformed markers be warnings by default or hard errors behind a strict mode?
- Should `Performance::measure` live under `performance`, `bench`, or another module?
- Should the canister helper be behind a feature flag such as `canister` to avoid IC CDK dependencies for host-only users?
- Should the first extended marker use positional columns with a version tag, or move to named fields for optional fields like `span_id` and `scope`?
